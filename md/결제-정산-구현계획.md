# 카드 결제 & 급여 자동 정산 구현 계획

## 전체 흐름 요약

```
사장님 카드 등록 (빌링키 발급)
       ↓
구직자 계좌 등록 (실명인증)
       ↓
퇴근 시 사장님에게 결제 요청 알림
       ↓
사장님이 앱에서 결제 승인 (빌링키로 카드 자동 청구)
       ↓
플랫폼 수수료 차감 (예: 3%)
       ↓
구직자 계좌로 순수 일당 이체 (지급대행)
```

---

## PG사 선택: 토스페이먼츠 (권장)

### 비교표

| | 토스페이먼츠 | 포트원(PortOne) | 나이스페이먼츠 |
|---|---|---|---|
| **빌링키 발급** | ✅ 자체 지원 | ✅ 다중 PG 연동 | ✅ 지원 |
| **지급대행(출금)** | ✅ 자체 API | ❌ 별도 계약 필요 | ❌ 별도 계약 필요 |
| **계좌 실명조회** | ✅ 자체 API | 별도 | 별도 |
| **API 문서 품질** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ |
| **수수료** | 카드 2.0~3.5% | PG별 상이 | 카드 2.5~3.5% |
| **스타트업 친화** | ✅ 빠른 심사 | ✅ | △ |

### 토스페이먼츠 선택 이유
- **카드 수납 + 계좌 출금을 단일 벤더**로 해결 가능
- **지급대행 API**로 구직자 계좌 직접 이체 지원
- **계좌 실명조회 API** 내장
- Next.js 공식 예제 제공, 국내 스타트업 표준 스택

### 필요 계약
1. 토스페이먼츠 사업자 계정 (사업자등록증 필요)
2. 전자금융업 등록 or 간편결제 이용 (규모에 따라 다름)
3. 지급대행 서비스 별도 신청 (B2B 계약)

## 보안 원칙 (PCI DSS 준수)

> **카드번호/CVV를 서버에 직접 저장하면 불법.**
> 토스페이먼츠 JS SDK가 카드 정보를 수집 → 자체 서버로 전송 → 빌링키만 우리 DB에 저장.

- 카드 원본 데이터: 토스페이먼츠 서버에만 존재
- 우리 DB에는 `billingKey` (토큰), 카드 마지막 4자리, 카드사명만 저장
- 계좌번호: AES-256 암호화 후 저장, 복호화는 서버 측에서만
- 모든 금융 API 호출: 서버 사이드 전용 (클라이언트에서 시크릿 키 노출 금지)

## DB 스키마 추가

```prisma
// 사장님 등록 카드 (빌링키)
model PaymentMethod {
  id          String   @id @default(cuid())
  userId      String
  user        User     @relation("EmployerPaymentMethods", fields: [userId], references: [id], onDelete: Cascade)

  billingKey  String   @unique  // 토스페이먼츠 빌링키 (토큰)
  cardCompany String            // "현대카드", "신한카드" 등
  cardLast4   String            // "1234"
  cardType    String?           // "신용", "체크"
  isDefault   Boolean  @default(true)

  createdAt   DateTime @default(now())
}

// 구직자 등록 계좌
model BankAccount {
  id            String   @id @default(cuid())
  userId        String
  user          User     @relation("WorkerBankAccounts", fields: [userId], references: [id], onDelete: Cascade)

  bankCode      String            // "004" (국민), "088" (신한) 등 금융결제원 코드
  bankName      String            // "국민은행"
  accountNumber String            // AES-256 암호화된 계좌번호
  accountHolder String            // 예금주명
  isVerified    Boolean  @default(false)  // 실명조회 통과 여부
  isDefault     Boolean  @default(true)

  createdAt     DateTime @default(now())
}

// 근무 확정 세션 (퇴근 처리된 단위)
model WorkSession {
  id          String   @id @default(cuid())
  jobId       String
  job         JobPosting @relation(fields: [jobId], references: [id])
  workerId    String
  worker      User     @relation("WorkerSessions", fields: [workerId], references: [id])
  employerId  String
  employer    User     @relation("EmployerSessions", fields: [employerId], references: [id])

  workDate    String   // "2025-04-01"
  payAmount   Int      // 해당 일의 일당
  status      String   @default("pending")
  // "pending"   → 퇴근 완료, 결제 대기
  // "requested" → 사장님에게 결제 요청 알림 발송됨
  // "paid"      → 결제 완료, 정산 대기
  // "settled"   → 구직자 계좌 입금 완료
  // "failed"    → 결제 또는 입금 실패

  payment     Payment?
  createdAt   DateTime @default(now())
}

// 결제 & 정산 원장
model Payment {
  id               String      @id @default(cuid())
  workSessionId    String      @unique
  workSession      WorkSession @relation(fields: [workSessionId], references: [id])

  // 금액 breakdown
  grossAmount      Int         // 총 결제 금액 (사장님이 낸 금액)
  platformFeeRate  Float       // 수수료율 (예: 0.03)
  platformFee      Int         // 플랫폼 수수료 = grossAmount * platformFeeRate
  pgFee            Int         // PG 수수료 (토스페이먼츠 청구분, 추후 정산용)
  netAmount        Int         // 구직자 수령액 = grossAmount - platformFee

  // 결제 (사장님 카드 청구)
  chargeStatus     String      @default("pending")  // "pending" | "success" | "failed"
  tossPaymentKey   String?     // 토스페이먼츠 paymentKey (결제 고유 식별자)
  chargedAt        DateTime?

  // 지급 (구직자 계좌 이체)
  payoutStatus     String      @default("pending")  // "pending" | "success" | "failed"
  tossPayoutId     String?     // 토스페이먼츠 지급대행 ID
  settledAt        DateTime?

  // 실패 처리
  failureCode      String?
  failureMessage   String?

  createdAt        DateTime    @default(now())
  updatedAt        DateTime    @updatedAt
}
```

User 모델에 relation 추가:
```prisma
paymentMethods PaymentMethod[] @relation("EmployerPaymentMethods")
bankAccounts   BankAccount[]   @relation("WorkerBankAccounts")
workSessions   WorkSession[]   @relation("WorkerSessions")
employerSessions WorkSession[] @relation("EmployerSessions")
```

## Phase 1: 사장님 카드 등록

### UX 플로우

```
/employer/settings → "카드 등록" 버튼
  → 토스페이먼츠 카드 등록 위젯 로드 (iframe)
  → 사장님이 카드 정보 입력 (토스페이먼츠 UI)
  → 토스페이먼츠 서버 → 빌링키 발급
  → 빌링키 + 카드 마지막 4자리 → 우리 서버 저장
  → 등록 완료 토스트
```

### API

```
POST /api/payment/card/register
  body: { billingKey, cardCompany, cardLast4, cardType }
  → PaymentMethod 저장

GET /api/payment/card
  → 등록된 카드 목록 (billingKey 제외, 카드사/마지막4자리만)

DELETE /api/payment/card/[id]
  → 카드 삭제
```

### 토스페이먼츠 빌링키 발급 방식

```
클라이언트: loadTossPayments(clientKey) → 카드 등록 위젯 렌더
  → 사장님이 카드 입력
  → 토스 서버에서 customerKey 매핑

서버: POST https://api.tosspayments.com/v1/billing/authorizations/card
  Authorization: Basic {base64(secretKey:)}
  body: { customerKey, authKey }
  → 응답: { billingKey, card: { company, number } }
  → billingKey를 PaymentMethod 테이블에 저장
```

## Phase 2: 구직자 계좌 등록

### UX 플로우

```
/worker/settings → "계좌 등록" 버튼
  → 은행 선택 (드롭다운, 금융결제원 기준 은행 목록)
  → 계좌번호 입력
  → "실명 확인" 버튼
    → POST /api/payment/account/verify (서버에서 토스페이먼츠 계좌 실명조회 API 호출)
    → 예금주명 확인 ("홍길동 명의의 국민은행 계좌입니다")
  → 구직자가 확인 → 계좌 저장
```

### API

```
POST /api/payment/account/verify
  body: { bankCode, accountNumber }
  → 토스페이먼츠 계좌 실명조회 API 호출
  → 응답: { holderName, isValid }

POST /api/payment/account/register
  body: { bankCode, bankName, accountNumber, accountHolder }
  → 계좌번호 AES-256 암호화 후 BankAccount 저장
  → isVerified: true 설정

GET /api/payment/account
  → 등록된 계좌 (계좌번호 마스킹: ***-****-1234 형태)
```

### 계좌번호 암호화

```typescript
// lib/crypto.ts
import crypto from "crypto"

const ENCRYPTION_KEY = process.env.ACCOUNT_ENCRYPTION_KEY! // 32바이트 hex
const IV_LENGTH = 16

export function encrypt(text: string): string {
  const iv = crypto.randomBytes(IV_LENGTH)
  const cipher = crypto.createCipheriv("aes-256-cbc",
    Buffer.from(ENCRYPTION_KEY, "hex"), iv)
  const encrypted = Buffer.concat([cipher.update(text), cipher.final()])
  return iv.toString("hex") + ":" + encrypted.toString("hex")
}

export function decrypt(text: string): string {
  const [ivHex, encryptedHex] = text.split(":")
  const iv = Buffer.from(ivHex, "hex")
  const encrypted = Buffer.from(encryptedHex, "hex")
  const decipher = crypto.createDecipheriv("aes-256-cbc",
    Buffer.from(ENCRYPTION_KEY, "hex"), iv)
  return Buffer.concat([decipher.update(encrypted), decipher.final()]).toString()
}
```

## Phase 3: 퇴근 처리 & 결제 요청

### UX 플로우 (사장님)

```
/employer (대시보드)
  → 오늘 출근 근무자 목록 표시
  → 오후 5시 이후 "퇴근 처리" 버튼 활성화
  → 사장님이 퇴근 근무자 체크박스 선택
  → "결제 요청" 버튼 클릭
    → 확인 다이얼로그: "박철수(130,000원), 김민수(130,000원) — 총 260,000원 결제하시겠어요?"
  → 확인 → POST /api/payment/request
    → WorkSession 레코드 생성
    → 알림(Notification) 생성: "결제를 완료해주세요"
  → 사장님에게 결제 확인 화면 이동
```

### 결제 확인 화면

```
/employer/payment/confirm
  → 결제 대상 목록
  → 등록된 카드 정보 (마지막 4자리)
  → 총 결제 금액
  → "지금 결제" 버튼 → POST /api/payment/charge
```

### API

```
POST /api/payment/request
  body: { workerIds: string[], jobId: string, workDate: string }
  → WorkSession 레코드 생성 (status: "requested")
  → Notification 생성 (사장님에게)
  → 응답: { workSessionIds, totalAmount }

POST /api/payment/charge
  body: { workSessionIds: string[] }
  → 각 WorkSession의 payAmount 합산
  → 사장님 billingKey로 토스페이먼츠 빌링 결제 API 호출
    POST https://api.tosspayments.com/v1/billing/{billingKey}
    body: { customerKey, amount, orderId, orderName }
  → 결제 성공 → Payment 레코드 업데이트 (chargeStatus: "success")
  → WorkSession status: "paid"
  → 정산 작업 큐에 추가 (또는 즉시 실행)
```

## Phase 4: 자동 정산 (구직자 계좌 입금)

### 정산 로직

```
grossAmount = 130,000원 (사장님 결제액)
platformFeeRate = 0.03 (3%)
platformFee = 3,900원
netAmount = 126,100원 (구직자 수령액)
```

> 수수료율은 환경변수(`PLATFORM_FEE_RATE`)로 관리하여 변경 용이

### 정산 실행 방식

**방식 A: 결제 직후 즉시 정산 (단순, 초기 권장)**
```
결제 성공 webhook 수신
  → 각 WorkSession별 구직자 계좌로 이체 API 호출
  → Payment.payoutStatus 업데이트
  → 구직자 Notification 생성: "130,000원이 입금되었습니다"
```

**방식 B: 배치 정산 (확장성, 추후 고려)**
```
매일 오후 11시 cron job
  → status: "paid"인 WorkSession 조회
  → 벌크 이체 (토스페이먼츠 지급대행 벌크 API)
  → 일괄 정산 처리
```

**초기 구현은 방식 A 권장.**

### 토스페이먼츠 지급대행 API (출금)

```
POST https://api.tosspayments.com/payouts/v1/bank-transfer/transfers
Authorization: Basic {base64(secretKey:)}
body: {
  amount: 126100,
  bankCode: "004",
  accountNumber: "1234567890",  // 복호화된 계좌번호
  holderName: "김민수",
  transferPurpose: "근로소득",
  idempotencyKey: payment.id    // 중복 이체 방지
}
→ 응답: { transferId, status }
```

### Webhook 처리

```
POST /api/payment/webhook
  → 토스페이먼츠 이벤트 수신
  → 서명 검증 (HMAC-SHA256)
  → 이벤트 타입별 처리:
    - PAYMENT_STATUS_CHANGED → Payment 상태 업데이트
    - TRANSFER_STATUS_CHANGED → Payout 상태 업데이트
    - 실패 시 → 사장님/관리자 알림, 재시도 로직
```

```
1. [인프라] 토스페이먼츠 테스트 계정 발급 + 환경변수 설정

6. [결제] 퇴근 처리 + 결제 요청 API

8. [정산] 결제 성공 후 즉시 지급대행 API 호출
9. [webhook] 토스페이먼츠 이벤트 수신 + 상태 동기화
10. [알림] 입금 완료 알림 연동
11. [테스트] 토스페이먼츠 테스트 모드로 전체 E2E 검증
12. [전환] 실서비스 전환 (토스 심사 통과 후)
```

| **전자금융업 등록** | 자금을 중간에 보유하고 송금하는 구조 → 규모에 따라 전자금융업자 등록 필요. 초기엔 토스페이먼츠 지급대행 서비스 이용으로 우회 가능 |
| **일용직 원천징수** | 일당 지급 시 3.3% 원천징수 의무 (이미 `/employer/tax`에 관련 기능 존재). 정산 금액에 세금 처리 로직 포함 고려 |
| **지급대행 계약** | 토스페이먼츠 지급대행은 별도 B2B 계약 필요, 심사 기간 1~2주 |
| **계좌번호 암호화 키 관리** | 환경변수 유출 시 모든 계좌번호 노출 → Vercel 환경변수 또는 AWS KMS 사용 권장 |
| **중복 이체 방지** | `idempotencyKey = payment.id` 로 동일 Payment에 대한 중복 이체 방지 필수 |
| **결제 실패 재시도** | 사장님 카드 한도/정지 시 재결제 플로우 필요 (카드 교체 → 재청구) |
