# 실서비스 구현 주의사항 보고서
### 일손매칭 — 급구 수준의 안정성을 위한 전체 체크리스트

> 처음 실서비스를 만든다면 "코드가 작동하는 것"과 "서비스가 안전하게 운영되는 것"은 완전히 다른 문제다.
> 이 문서는 출시 전 반드시 검토해야 할 모든 항목을 체크리스트 형태로 정리한다.

---

## 🔐 1. 보안 (Security)

### 1-1. 인증 & 인가

- [ ] **모든 API 라우트에 인증 검사** — `auth()` 호출 없이 데이터를 반환하는 라우트 없어야 함
- [ ] **역할(Role) 검사를 클라이언트에만 의존하지 않기** — "사장님만" 버튼을 숨기는 것만으로는 부족, 서버에서도 `role !== "employer"` 체크 필수
- [ ] **본인 소유 데이터만 접근 가능** — `/api/employer/jobs`에서 `employerId === session.user.id` 확인 필수 (다른 사장님 공고를 수정/삭제 못하게)
- [ ] **세션 토큰 만료 처리** — NextAuth `session.expires` 지난 요청 거부, 프론트에서 자동 재로그인 처리
- [ ] **소셜 로그인 후 프로필 미완성 유저 접근 제한** — `isProfileComplete === false`이면 대시보드 접근 차단, 회원가입 페이지로 강제 이동

### 1-2. API 보안

- [ ] **Rate Limiting (요청 제한)** — 로그인 API는 동일 IP에서 분당 10회 이상 요청 시 차단 (Vercel Edge Config 또는 upstash/ratelimit 사용)
- [ ] **지원하기 중복 요청 방지** — DB `@@unique([workerId, jobId])` 이미 있음 → 프론트에서 추가로 버튼 비활성화 처리
- [ ] **CSRF 방어** — NextAuth는 기본으로 처리하지만, 커스텀 API 라우트에서 `Origin` / `Referer` 헤더 검증
- [ ] **입력값 검증은 서버에서** — 프론트 validation만으로 충분하지 않음, 모든 POST body를 서버에서 zod로 재검증
- [ ] **SQL Injection 방지** — Prisma 사용 시 자동 방어되지만, 절대 raw query에 사용자 입력 직접 삽입 금지

```typescript
// 위험 (절대 금지)
await prisma.$queryRaw`SELECT * FROM users WHERE name = '${userInput}'`

// 안전
await prisma.$queryRaw`SELECT * FROM users WHERE name = ${userInput}`
// 또는 parameterized query 사용
```

- [ ] **HTTP 응답에 민감정보 포함 금지** — 에러 메시지에 DB 구조, stack trace, 내부 경로 노출 금지

```typescript
// 위험
return NextResponse.json({ error: err.message }) // "PrismaClientKnownRequestError: ..."

// 안전
console.error(err) // 서버 로그에만
return NextResponse.json({ error: "오류가 발생했습니다" }, { status: 500 })
```

### 1-3. 금융 보안

- [ ] **빌링키 서버 외부 노출 금지** — 클라이언트 응답에 `billingKey` 절대 포함 금지
- [ ] **계좌번호 AES-256 암호화 저장** — 평문 저장 절대 금지
- [ ] **암호화 키 환경변수 관리** — `.env` 파일에 저장, `.gitignore`에 반드시 포함
- [ ] **웹훅 서명 검증** — 토스페이먼츠 웹훅 수신 시 `HMAC-SHA256` 서명 반드시 검증 (미검증 시 위조 요청 처리 가능)
- [ ] **중복 결제 방지 (Idempotency Key)** — 같은 WorkSession에 두 번 청구되지 않도록 `idempotencyKey` 적용
- [ ] **결제 금액 서버에서 계산** — 클라이언트에서 받은 금액을 그대로 청구하면 안 됨, 반드시 DB의 `payAmount`로 서버에서 계산

```typescript
// 위험
const { amount } = await req.json() // 클라이언트가 보낸 금액 신뢰
await toss.charge(billingKey, amount)

// 안전
const session = await prisma.workSession.findUnique({ where: { id: workSessionId } })
await toss.charge(billingKey, session.payAmount) // DB 원본 금액으로
```

### 1-4. 환경변수 관리

- [ ] `.env.local` 파일 `.gitignore`에 포함 확인 (GitHub에 절대 올라가면 안 됨)
- [ ] Vercel 환경변수에 Production/Preview/Development 구분 설정
- [ ] 민감 키 목록 (`TOSS_SECRET_KEY`, `ACCOUNT_ENCRYPTION_KEY`, `NEXTAUTH_SECRET`) 별도 문서화
- [ ] 팀원 퇴사 시 모든 API 키 즉시 재발급 프로세스 수립

## 📋 2. 법률 & 규제 (Legal & Compliance)

### 2-1. 개인정보보호법 (PIPA)

- [ ] **개인정보처리방침 작성 및 서비스 내 게시** (법적 의무) — 수집 항목, 목적, 보유 기간, 제3자 제공 여부 명시
- [ ] **회원 가입 시 개인정보 수집·이용 동의 체크박스** — 필수 동의 / 선택 동의 분리 (마케팅 수신 등)
- [ ] **만 14세 미만 가입 차단** — 법정대리인 동의 없이 미성년자 서비스 불가
- [ ] **개인정보 보유 기간 정책** — 탈퇴 후 즉시 삭제 또는 법정 보유 기간(3년) 후 삭제
- [ ] **개인정보 파기 절차** — 계정 삭제 시 관련 데이터 anonymize 또는 delete cascade 처리
- [ ] **개인정보 유출 시 72시간 내 신고 의무** — 한국인터넷진흥원(KISA) 신고 프로세스 사전 파악
- [ ] **개인정보 처리 위탁 계약** — 토스페이먼츠, Neon DB, Vercel 등 제3자에게 데이터가 넘어가는 경우 위탁 계약서 필요

### 2-2. 전자금융거래법

- [ ] **자금이체업 등록 여부 검토** — 플랫폼이 사장님 돈을 받아 구직자에게 넘기는 구조 → 규모에 따라 전자자금이체업 등록 필요
- [ ] **소액 수준에선 토스페이먼츠 지급대행 활용** — 직접 자금을 보유하지 않고 PG사를 통해 처리하면 전자금융업 등록 우회 가능 (초기 적합)
- [ ] **거래 내역 5년 보관 의무** — 결제/정산 로그는 법적으로 5년간 보관 필요

### 2-3. 근로기준법

- [ ] **일용직 근로계약서 제공** — 일용직도 근로계약서 작성 권장 (법적 분쟁 방지), 앱 내 무료 계약서 생성 기능 고려
- [ ] **최저임금 위반 공고 차단** — 공고 등록 시 `payAmount`가 최저임금 미만이면 경고 또는 등록 차단 (2025년 최저시급 10,030원)
- [ ] **주 52시간 초과 근무 안내** — 연속 근무 일수가 법적 한도 초과 시 사장님에게 안내
- [ ] **산업재해 발생 시 신고 의무** — 이용약관에 사고 발생 시 플랫폼 책임 범위 명시

### 2-4. 사업자 등록 & 서비스 운영

- [ ] **사업자등록** (필수) — 수수료 수입 발생 시 사업자 없으면 불법 영업
- [ ] **통신판매업 신고** — 온라인으로 서비스 중개 시 필요
- [ ] **이용약관 작성** — 서비스 이용 규칙, 금지행위, 계정 정지 기준 명시
- [ ] **환불 정책 명시** — 결제 후 취소 시 수수료 처리 방침

## 🏗️ 4. 인프라 & 배포

### 4-1. 데이터베이스

- [ ] **자동 백업 설정** — Neon DB 자동 백업 활성화 (Point-in-time recovery), 최소 7일치 보관
- [ ] **DB 마이그레이션 전략** — `prisma migrate deploy` (production 명령) 와 `prisma migrate dev` (개발 명령) 구분해서 사용
- [ ] **마이그레이션 롤백 계획** — 배포 후 문제 발생 시 이전 스키마로 되돌리는 방법 사전 수립
- [ ] **Connection Pool 설정** — Vercel 서버리스 환경에서 DB 연결 폭증 방지 (`DATABASE_URL`에 `?pgbouncer=true&connection_limit=1` 설정)
- [ ] **트랜잭션 사용** — 결제 + WorkSession 상태 변경 같은 "함께 성공하거나 함께 실패해야 하는" 로직에 반드시 트랜잭션 사용

```typescript
// 결제 + 상태 업데이트 원자적 처리
await prisma.$transaction([
  prisma.payment.update({ where: { id }, data: { chargeStatus: "success" } }),
  prisma.workSession.update({ where: { id }, data: { status: "paid" } }),
])
```

- [ ] **N+1 쿼리 문제 방지** — 목록 조회 시 `include`로 연관 데이터를 한 번에 가져오기 (루프 안에서 개별 쿼리 금지)

### 4-2. 배포 환경

- [ ] **Production / Staging / Development 환경 분리** — Production DB에 개발 중 테스트 데이터 절대 들어가면 안 됨
- [ ] **Staging 환경** — 실 배포 전 테스트할 별도 환경 (Vercel Preview 활용)
- [ ] **환경별 토스페이먼츠 키 분리** — 테스트 키(`test_sk_...`) 와 라이브 키(`live_sk_...`) 혼용 금지
- [ ] **빌드 실패 시 자동 롤백** — Vercel은 자동 처리, 커스텀 서버라면 이전 버전으로 즉시 복구 전략 필요
- [ ] **도메인 HTTPS 설정** — 커스텀 도메인 사용 시 SSL 인증서 자동 갱신 확인

### 4-3. 모니터링 & 알람

- [ ] **에러 추적 도구 설정** — Sentry 또는 Vercel Error Tracking 연동 → 에러 발생 시 즉시 Slack/이메일 알림
- [ ] **로그 수집** — Vercel Logs 또는 Axiom 연동 → API 응답 시간, 에러율 모니터링
- [ ] **업타임 모니터링** — UptimeRobot (무료) 설정 → 서비스 다운 시 즉시 SMS 알림
- [ ] **DB 용량 모니터링** — Neon 대시보드에서 용량 알람 설정
- [ ] **결제 실패율 대시보드** — 토스페이먼츠 대시보드에서 결제 성공률 주기적 확인

## 💾 5. 데이터 정합성

- [ ] **금액 계산은 정수(Int)로** — 부동소수점 오차 방지, `130000.0000000001` 같은 오류 발생 가능, DB에 `Int` 타입 사용
- [ ] **결제 상태 머신 명확화** — WorkSession status가 `pending → requested → paid → settled` 순서대로만 변할 수 있도록 서버에서 강제
- [ ] **동시성 처리** — 같은 공고에 두 명이 동시에 마지막 1자리에 지원할 때 처리 (DB `unique constraint` + 트랜잭션으로 방어)
- [ ] **소프트 딜리트 전략** — 공고나 계정을 실제로 삭제하지 않고 `status: "deleted"` 처리 → 분쟁 시 데이터 복구 가능
- [ ] **감사 로그 (Audit Log)** — 결제, 계정 정지, 데이터 변경 등 중요 이벤트는 별도 로그 테이블에 who/what/when 기록

```prisma
model JobPosting {

  @@index([status, createdAt])

  @@index([location])
}
```

## 🧪 7. 테스트

### 7-1. 반드시 테스트해야 할 시나리오

**인증 플로우:**
- [ ] 소셜 로그인 → 프로필 미완성 → 회원가입 페이지 이동
- [ ] 로그아웃 상태에서 보호된 페이지 직접 URL 입력 시 로그인 페이지로 이동
- [ ] 구직자 계정으로 사장님 API 호출 시 403 반환

**공고 플로우:**
- [ ] 사장님이 공고 등록 → 구직자 공고 목록에 즉시 표시
- [ ] 정원이 찬 공고에 추가 지원 시 차단
- [ ] 이미 지원한 공고에 재지원 시 "이미 지원한 공고" 에러

**결제 플로우:**
- [ ] 카드 한도 초과 시 실패 처리 → 사장님 알림
- [ ] 결제 성공 후 구직자 계좌 입금 확인
- [ ] 동일 WorkSession 두 번 결제 시도 시 중복 방지
- [ ] 웹훅 서명 위조 시 거부

**에러 케이스:**
- [ ] DB 연결 실패 시 사용자에게 적절한 에러 메시지
- [ ] 토스 API 타임아웃 시 재시도 또는 실패 처리
- [ ] 잘못된 계좌번호로 이체 시도 시 처리

### 7-2. 테스트 환경

- [ ] **토스페이먼츠 테스트 모드** — 실제 돈이 오가지 않는 테스트 카드 번호 사용
- [ ] **테스트 계정 세트** — 사장님 계정 1개, 구직자 계정 3개를 테스트용으로 항상 유지
- [ ] **Playwright E2E 테스트** — 핵심 플로우(공고 등록 → 지원 → 결제) 자동화 테스트 (선택 사항이지만 강력 권장)

## 🚨 8. 에러 처리 & 사용자 경험

- [ ] **모든 API 에러를 사용자 친화적 메시지로** — "Internal Server Error" 대신 "잠시 후 다시 시도해주세요"

### 15-1. OWASP Top 10 대응 (2023 기준)

#### A01 — 취약한 접근 제어 (Broken Access Control)

- [ ] **수평적 권한 상승 방지** — 구직자 A가 구직자 B의 지원 내역, 계좌 정보에 접근하지 못하도록 모든 DB 조회에 `userId` 필터 강제

```typescript
// 위험: id만으로 조회 → 타인 데이터 접근 가능
const account = await prisma.bankAccount.findUnique({ where: { id: params.id } })

// 안전: userId 교차 검증 필수
const account = await prisma.bankAccount.findFirst({
  where: { id: params.id, userId: session.user.id }
})
if (!account) return NextResponse.json({ error: "없는 리소스입니다" }, { status: 404 })
// 403이 아닌 404 반환 → 리소스 존재 여부 자체를 숨김 (정보 노출 방지)
```

- [ ] **수직적 권한 상승 방지** — 구직자가 `role: "employer"` 로 세션을 위조해도 서버에서 DB 원본 role을 재조회하는 방어

```typescript
// 세션의 role을 신뢰하지 말고 DB에서 재확인
const dbUser = await prisma.user.findUnique({ where: { id: session.user.id } })
if (dbUser?.role !== "employer") return 403
```

- [ ] **직접 객체 참조(IDOR) 방어** — URL의 `/api/jobs/[id]` 에서 `id`를 숫자 예측으로 바꿔 타인 공고 수정 시도 차단
- [ ] **CUID/UUID 사용** — Prisma 기본 `cuid()` 사용 중 → 순차 ID(`1, 2, 3`) 대신 예측 불가 ID 자동 적용됨 ✅

#### A02 — 암호화 실패 (Cryptographic Failures)

- [ ] **전송 중 암호화** — HTTPS(TLS 1.2+) 강제, HTTP 요청은 301로 HTTPS 리다이렉트
- [ ] **저장 데이터 암호화 등급**

| 데이터 | 처리 방식 | 비고 |
|--------|----------|------|
| 비밀번호 | 해당 없음 (소셜 로그인) | bcrypt 불필요 |
| 계좌번호 | AES-256-CBC 암호화 후 저장 | 복호화 필요한 데이터 |
| 카드번호 | 저장 금지 (빌링키만 저장) | PCI DSS 필수 |
| 전화번호 | 평문 저장 가능 (필요 시 암호화) | 검색 필요성 고려 |
| 이름 | 평문 저장 가능 | |
| 이메일 | 평문 저장 가능 | |

- [ ] **암호화 알고리즘 선택 기준**
  - AES-256-CBC with random IV: 계좌번호 등 복호화 필요한 데이터
  - SHA-256 HMAC: 웹훅 서명 검증
  - 절대 사용 금지: MD5, SHA-1 (이미 크랙됨), DES, RC4
- [ ] **IV(초기화 벡터) 재사용 금지** — 암호화마다 `crypto.randomBytes(16)` 으로 새 IV 생성, IV를 암호문과 함께 저장

```typescript
// 올바른 AES-256-CBC 구현
import crypto from "crypto"

const ALGO = "aes-256-cbc"
const KEY = Buffer.from(process.env.ACCOUNT_ENCRYPTION_KEY!, "hex") // 32 bytes

export function encrypt(plain: string): string {
  const iv = crypto.randomBytes(16) // 매번 새로운 IV
  const cipher = crypto.createCipheriv(ALGO, KEY, iv)
  const encrypted = Buffer.concat([cipher.update(plain, "utf8"), cipher.final()])
  // IV:암호문 형태로 저장 (IV는 공개해도 됨, 비밀은 KEY)
  return iv.toString("hex") + ":" + encrypted.toString("hex")
}

export function decrypt(stored: string): string {
  const [ivHex, encHex] = stored.split(":")
  const iv = Buffer.from(ivHex, "hex")
  const encrypted = Buffer.from(encHex, "hex")
  const decipher = crypto.createDecipheriv(ALGO, KEY, iv)
  return Buffer.concat([decipher.update(encrypted), decipher.final()]).toString("utf8")
}
```

#### A03 — 인젝션 (Injection)

- [ ] **SQL Injection** — Prisma ORM 사용으로 기본 방어, `$queryRaw` 사용 시 반드시 tagged template literal 사용 (위 예시 참고)
- [ ] **NoSQL Injection** — 해당 없음 (PostgreSQL 사용)
- [ ] **XSS (Cross-Site Scripting)** — 사용자 입력을 `dangerouslySetInnerHTML`로 렌더링 금지, React 기본 이스케이핑 활용

```typescript
// 위험 (절대 금지)
<div dangerouslySetInnerHTML={{ __html: job.description }} />

// 안전 (React 기본 이스케이핑)
<p>{job.description}</p>

// 사용자 입력을 HTML로 표시해야 한다면: DOMPurify 라이브러리 사용
import DOMPurify from "dompurify"
<div dangerouslySetInnerHTML={{ __html: DOMPurify.sanitize(job.description) }} />
```

- [ ] **저장형 XSS 방지** — DB 저장 전 사용자 입력에서 `<script>`, `javascript:`, `onload=` 등 필터링 또는 서버에서 HTML 이스케이프

```typescript
// 저장 전 기본 위험 문자 제거
function sanitizeText(input: string): string {
  return input
    .replace(/</g, "&lt;")
    .replace(/>/g, "&gt;")
    .replace(/"/g, "&quot;")
    .replace(/'/g, "&#x27;")
    .trim()
}
```

- [ ] **Open Redirect 방지** — `callbackUrl` 파라미터를 그대로 리다이렉트하면 피싱 사이트로 연결 가능

```typescript
// 위험
const callbackUrl = searchParams.get("callbackUrl")
redirect(callbackUrl) // 외부 URL로 리다이렉트 가능

// 안전: 자사 도메인만 허용
const allowedPaths = ["/jobs", "/employer", "/worker"]
const callbackUrl = searchParams.get("callbackUrl") ?? "/"
const safeUrl = allowedPaths.includes(callbackUrl) ? callbackUrl : "/"
redirect(safeUrl)
```

#### A04 — 안전하지 않은 설계 (Insecure Design)

- [ ] **Fail-Safe Default** — 오류 발생 시 기본적으로 "거부"하도록 설계 (오류 시 접근 허용이 아닌 차단)
- [ ] **최소 권한 원칙** — DB 계정에 SELECT/INSERT/UPDATE만 부여, DROP/TRUNCATE 권한 제거
- [ ] **심층 방어(Defense in Depth)** — 프론트 검증 + 서버 검증 + DB 제약조건 3중 방어

```
[프론트] zod 스키마 검증 (UX 즉시 피드백)
     ↓
[서버] zod 재검증 + 비즈니스 로직 검증
     ↓
[DB] NOT NULL, UNIQUE, CHECK 제약조건
```

```typescript
function isSafeUrl(url: string): boolean {
  try {
    const parsed = new URL(url)
    // 내부 IP 범위 차단
    const blockedHosts = ["localhost", "127.0.0.1", "0.0.0.0", "169.254.169.254"]
    if (blockedHosts.includes(parsed.hostname)) return false
    // HTTPS만 허용
    if (parsed.protocol !== "https:") return false
    return true
  } catch {
    return false
  }
}
```

```typescript
// 결제 요청 위변조 방지
function createPaymentSignature(workSessionId: string, amount: number): string {
  const timestamp = Date.now()
  const payload = `${workSessionId}:${amount}:${timestamp}`
  return crypto
    .createHmac("sha256", process.env.PAYMENT_SIGNING_KEY!)
    .update(payload)
    .digest("hex") + `:${timestamp}`
}

function verifyPaymentSignature(sig: string, workSessionId: string, amount: number): boolean {
  const [hash, timestamp] = sig.split(":")
  // 5분 이내 요청만 유효 (리플레이 공격 방지)
  if (Date.now() - Number(timestamp) > 5 * 60 * 1000) return false
  const expected = createPaymentSignature(workSessionId, amount).split(":")[0]
  return crypto.timingSafeEqual(Buffer.from(hash), Buffer.from(expected))
}
```

## 💡 처음 서비스를 만드는 분께 — 핵심 3가지

> **1. "작동하는 코드"와 "안전한 서비스"는 다르다**
> 로컬에서 잘 돌아간다고 끝이 아니다. 실제 사용자 데이터, 실제 돈이 들어오는 순간부터 보안과 법규를 무시할 수 없다.

> **2. 처음부터 완벽할 필요는 없지만, 치명적인 것부터 막아라**
> 결제 중복, 계좌번호 평문 저장, 인증 없는 API — 이 세 가지만 막아도 큰 사고를 방지할 수 있다.

> **3. 모니터링 없는 서비스는 눈을 감고 운전하는 것이다**
> Sentry + 업타임 모니터링 + DB 백업. 이 세 가지는 출시 전날에라도 반드시 설정해라. 장애는 반드시 온다 — 문제는 얼마나 빨리 아느냐다.
