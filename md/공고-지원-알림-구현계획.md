# 공고 노출 & 지원 알림 구현 계획

## 현황 분석

### Feature 1: 사장님 공고 → 구직자 공고 페이지 노출
**현재 상태: 파이프라인 이미 구축됨 ✅**

```
사장님 공고 등록 (/employer/post)
  → POST /api/jobs (route.ts)
  → DB 저장 (JobPosting 테이블, status: "active")
  → GET /api/jobs (route.ts)
  → /jobs 페이지가 fetch → 화면 표시
```

실제로 코드가 모두 연결되어 있음. **다만 실제로 동작하는지 E2E 검증이 필요**.

**잠재적 문제점 체크리스트:**
- [ ] `/employer/post` 제출 후 DB에 실제로 저장되는지 확인 (Neon 콘솔 or `prisma studio`)
- [ ] `/jobs` 페이지에서 실시간으로 갱신되는지 확인 (페이지 새로고침 시 반영 여부)
- [ ] `status: "active"` 기본값이 실제로 적용되는지 확인

---

### Feature 2: 구직자 지원 → 사장님 알림
**현재 상태: 미구현 ❌**

지원 저장(`POST /api/jobs/[id]/apply`) 이후 알림을 전달하는 로직 전무.

## 구현 계획

### Phase 1: Feature 1 검증 (공고 연동)

> 실제 구현 없이 동작 확인만 필요

**확인 순서:**
1. 사장님 계정으로 `/employer/post` 에서 공고 등록

3. 구직자 계정으로 로그인 → 지원하기 버튼 클릭 → DB에 Application 레코드 생성 확인

문제 발생 시:
- `DATABASE_URL` 환경변수 연결 상태 점검
- `prisma db push` 로 스키마 동기화 여부 확인

### Phase 2: 알림 시스템 구현

#### 2-1. DB 스키마 추가 (`prisma/schema.prisma`)

```prisma
model Notification {
  id         String   @id @default(cuid())
  userId     String                          // 알림 수신자 (사장님)
  user       User     @relation(fields: [userId], references: [id], onDelete: Cascade)

  type       String                          // "new_application"
  title      String
  body       String
  isRead     Boolean  @default(false)
  metadata   Json?                           // { jobId, jobTitle, workerName, applicationId }

  createdAt  DateTime @default(now())
}
```

User 모델에 relation 추가:
```prisma
notifications Notification[] @relation("UserNotifications")
```

마이그레이션:
```bash
npx prisma migrate dev --name add-notification
```

---

#### 2-2. 지원 시 알림 생성 (`/api/jobs/[id]/apply/route.ts`)

`application.create` 이후 아래 로직 추가:

```
1. job.employerId 조회 (지원한 공고의 사장님 ID)
2. 지원한 구직자의 name 조회
3. Notification.create({
     userId: job.employerId,
     type: "new_application",
     title: "새 지원자가 있어요",
     body: `${workerName}님이 [${job.companyName}] 공고에 지원했습니다`,
     metadata: { jobId, applicationId, workerName }
   })
```

---

#### 2-3. 알림 조회 API (`/api/notifications/route.ts`)

```
GET /api/notifications
  - 인증 필요 (사장님)
  - 본인의 알림 목록 반환 (최신순, 최대 50개)
  - 응답: { notifications: Notification[], unreadCount: number }

PATCH /api/notifications/[id]/read
  - 특정 알림 읽음 처리

PATCH /api/notifications/read-all
  - 전체 읽음 처리
```

#### 2-4. 사장님 대시보드 알림 UI (`/app/employer/page.tsx`)

**알림 벨 아이콘 (헤더 우상단):**
- 미읽음 알림 수 뱃지 표시
- 클릭 시 알림 드롭다운 또는 알림 페이지로 이동

**알림 목록 표시 방식 (선택지):**
| 방식 | 장점 | 단점 |
|------|------|------|
| 폴링 (setInterval, 30초) | 구현 단순 | 약간의 딜레이, 불필요한 요청 |
| SSE (Server-Sent Events) | 실시간, 단방향이라 구현 무난 | 서버리스 환경에서 연결 유지 제한 |
| Vercel KV + 폴링 | 확장성 좋음 | 추가 인프라 필요 |

**권장: 폴링 방식 (30초 간격)**
- 현재 Next.js + Vercel 환경에서 가장 안정적
- `useEffect` + `setInterval` + `fetch('/api/notifications')`

## 보류/추후 고려 사항

- **Push 알림 (Web Push / FCM):** 브라우저 닫혀도 알림 수신 가능하나, 별도 서비스워커 + 인프라 필요
- **카카오 알림톡:** 사장님 전화번호로 카카오 메시지 발송 — 실제 서비스 단계에서 고려
- **지원자 목록 페이지 (`/employer/applicants`):** 알림 클릭 시 이동할 상세 페이지 (현재 미구현)
- **알림 설정:** 사장님이 특정 알림을 끌 수 있는 토글 (추후)
